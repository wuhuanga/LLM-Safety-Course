#### 1. 整体效果

- **批量编辑规模**：使用 MEMIT 算法一次性向 Qwen2.5-0.5B 模型注入了 **500 条知识**（基于 ZsRE 数据集的问答对）。
- **资源消耗**：编辑耗时 **24.4 分钟**，峰值 GPU 显存 **3437 MB**（约 3.4 GB），表明 MEMIT 在消费级 GPU（如 RTX 3090/4090）上可顺畅运行。

#### 2. 与 ROME 单条编辑的对比

- **ROME（10条）** 实现了 100% ES，但 NS 仅 65.5%，说明单条编辑虽然精准，却容易破坏少量邻近事实。
- **MEMIT（500条）** 以牺牲 ES（30.2%）换取了极高的局部性（97.3%），体现了批量编辑的“知识隔离”优势。

### 2、MEMIT 批量编辑分析

#### 📌 类型 1：完全失败（rewrite_acc = 0.0，rephrase_acc = 0.0）

**占比**：在所给片段中约占 **35%~40%**（例如 case_id 0/1/2/3 下的多个记录，以及许多后续 case_id 中 `[0.0]` 的条目）。

**特点**：

- 编辑后模型对原始 prompt 和所有改写 prompt 均未输出目标答案。
- `locality_acc` 几乎均为 1.0，说明编辑没有破坏其他知识，但也没有成功写入任何新事实。

**典型代表**（摘自数据）：

```text
{"case_id": 2, "rewrite_acc": [0.0], "rephrase_acc": [0.0], "locality_acc": [1.0]}
{"case_id": 1, "rewrite_acc": [0.0], "rephrase_acc": [0.0], "locality_acc": [1.0]}
```

**可能原因**：

- **知识冲突**：新事实与模型原有的强关联知识矛盾过大，MEMIT 的参数更新被正则项抑制。
- **主体（subject）定位失败**：ZsRE 数据集中某些问题的 `subject` 提取不正确（例如 `"What is the capital of ..."` 可能提取了 `"capital"` 而非国家名），导致编辑了错误位置。
- **模型容量饱和**：0.5B 模型在批量编辑 500 条时，部分知识无法被有效编码。

#### 📌 类型 2：部分成功（rewrite_acc ∈ (0, 0.5)，rephrase_acc 相似或更低）

**占比**：约占 **40%~50%**。

**特点**：

- 编辑对部分测试 prompt 生效（例如 3 个改写中只成功了 1 个，得分为 0.333）。
- 泛化能力弱：rephrase_acc 几乎等于 rewrite_acc，说明成功案例也几乎过拟合原始表述。

**典型代表**：

```text
{"case_id": 0, "rewrite_acc": [0.3333333333333333], "rephrase_acc": [0.3333333333333333], "locality_acc": [1.0]}
{"case_id": 4, "rewrite_acc": [0.3333333333333333], "rephrase_acc": [0.3333333333333333], "locality_acc": [1.0]}
```

**可能原因**：

- MEMIT 默认的 `layers=[9,10,11,12,13]` 可能不是该模型存储事实的最佳层，导致部分知识写入强度不足。
- 学习率（`v_lr=0.5`）或梯度步数（`v_num_grad_steps=20`）偏低，未能完全优化目标。

#### 📌 类型 3：泛化失败（rewrite_acc > 0，但 rephrase_acc 明显更低）

**出现较少**，例如：

```text
{"case_id": 3, "rewrite_acc": [0.5], "rephrase_acc": [0.3333333333333333], ...}
```

这表明编辑对原始 prompt 有效，但无法推广到同义改写，说明编辑过于“死板”。

**原因**：

- 编辑时只使用了原始 prompt 的损失，未加入 rephrase 的监督信号（EasyEdit 默认不要求提供 rephrase 进行训练，仅评估时使用）。
- 模型记住了特定词汇组合，而非抽象事实。

#### 📌 类型 4：局部性受损（locality_acc < 1.0）

**占比极低**（在所给片段中仅发现几例）：

```text
{"case_id": 5, "rewrite_acc": [0.2], "rephrase_acc": [0.2], "locality_acc": [0.8]}
{"case_id": 3, "rewrite_acc": [0.4], "rephrase_acc": [0.8], "locality_acc": [0.6666666666666666]}
{"case_id": 8, "rewrite_acc": [0.25], "rephrase_acc": [0.25], "locality_acc": [0.5]}
```

**分析**：

- `locality_acc` 从 1.0 降到 0.5~0.8，意味着编辑破坏了约 20%~50% 的无关事实。
- 这些案例通常 `rewrite_acc` 也较低，说明 MEMIT 在某些样本上“乱更新”，干扰了原本正确的知识。
- 可能原因是这些样本的 `subject` 过于通用（如 `"country"`、`"person"`），导致参数更新波及范围过大。

| 原因类别         | 具体描述                                                     | 影响程度 |
| :--------------- | :----------------------------------------------------------- | :------- |
| **知识容量**     | Qwen2.5-0.5B 参数量小，难以一次性吸收 500 条新事实           | 高       |
| **超参数次优**   | 编辑层范围、学习率、梯度步数未针对 0.5B 模型调优             | 中       |
| **subject 噪声** | ZsRE 数据集中自动抽取的 subject 存在错误或不精确             | 中       |
| **事实冲突**     | 多条新知识映射到相似的神经元区域，相互覆盖                   | 中       |
| **评估偏差**     | 部分正确回答因格式不完全匹配被判定为失败（如输出完整句子但 target_new 仅为关键词） | 低       |

3. **原因推测**：

- 或者采用基于指令的编辑方法（如 InstructEdit）明确指定语言属性。
